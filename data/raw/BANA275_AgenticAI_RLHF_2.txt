Why Not Just Fine-Tune on Good Data?
Supervised fine-tuning (SFT) gets you 80% there — but the last 20% requires RLHF

Imitation ≠ Alignment
SFT teaches the model to copy human writing. But copying style doesn't mean understanding what makes a response actually helpful vs. harmful.

Preferences can't be demonstrated
Some qualities (helpfulness, safety, honesty) are easier to judge than to demonstrate. Humans rank outputs more reliably than they write perfect outputs.

Distribution of quality matters
SFT treats all training examples equally. RLHF lets you say "this response is better than that one" — capturing subtle quality gradients.

Safety requires boundaries
You can't enumerate every unsafe behavior in training data. RLHF teaches boundaries by showing the model what humans reject.
SFT teaches imitation. RLHF teaches judgment.
5
The RLHF Pipeline
Three stages to make models follow human intent

1. SFT — Supervised Fine-Tuning
Train on high-quality human demonstrations. The model learns to imitate expert behavior and produce well-formatted responses.

2. RM — Reward Model
Learn human preferences from ranked outputs. Humans compare response pairs; the reward model learns to predict which they'd prefer.

3. PPO — Policy Optimization
Optimize the model to maximize human-aligned reward while staying close to the original model via KL penalty.
6
The Math Behind RLHF
Two key equations — the reward model and the policy objective

REWARD MODEL LOSS
L(θ) = − E [ log σ( r_θ(x, y_w) − r_θ(x, y_l) ) ]
r_θ(x, y)  =  reward model's score for response y given prompt x
y_w, y_l   =  the preferred (winner) and rejected (loser) response
σ           =  sigmoid — converts score difference into probability
Goal: learn to assign higher scores to responses humans prefer

PPO POLICY OBJECTIVE
J(π) = E [ r_θ(x, y) − β · KL( π(y|x) || π_ref(y|x) ) ]
r_θ(x, y)     =  reward model score — "how good is this response?"
KL(π || π_ref) =  divergence from original — "how far did we drift?"
β              =  tradeoff: too low → reward hacking, too high → no learning
Goal: maximize reward while staying close to the reference model

What Can Go Wrong
Reward Hacking: Model finds shortcuts to high reward (Verbose but empty responses)
Sycophancy: Model agrees rather than being truthful ("You're absolutely right!" when wrong)
Mode Collapse: Model becomes repetitive (Every answer starts with "Great question!")

Modern Alternatives to RLHF

DPO (Direct Preference Optimization)
Skip the reward model entirely. Train directly on human preference pairs. Simpler, cheaper, increasingly popular.
Used by: Meta (Llama), Mistral

Constitutional AI / RLAIF
Use AI feedback instead of human feedback. Define principles, let the model self-critique.
Used by: Anthropic (Claude)

RLHF + Outcome-Based Rewards
Reward based on task outcomes (code passes tests, math correct) rather than style preferences.
Used by: OpenAI (o1, o3), DeepSeek (R1)

Pattern 1: ReAct
Reasoning + Acting — the foundational agent pattern (Yao et al., 2023)

EXAMPLE TRACE
User: What's the weather in Irvine?
Thought: I need to look up current weather.
Action: call weather_api(city='Irvine')
Observation: 72°F, sunny, low humidity
Thought: I have the info. Respond to user.
Answer: It's 72°F and sunny in Irvine!
How It Works
•  Think before acting
•  Each step is traceable
•  Observations feed back
•  Loop until done

Think → Act → Observe
→ Repeat

Pattern 3: Reflection
The agent critiques its own output and iterates — a quality amplifier

EXAMPLE TRACE
User: Write a product description for our new headphones.
Generate: "Experience crystal-clear audio with our latest wireless..."
Critique: "Too generic. No differentiators. No specific specs."
Refine: "The AX-7 delivers 40-hour battery life with adaptive noise cancellation..."
When to Use
•  Code generation (run tests, fix failures)
•  Writing tasks (draft → critique → polish)
•  Data extraction (extract → validate → re-extract)
Watch Out For
•  Infinite critique loops (set max iterations)
•  Over-polishing (diminishing returns after 2-3)
•  Cost multiplication (each round = API call)
21
Pattern 4: Planning Strategies
How agents decide what to do — from simple to sophisticated

Chain-of-Thought (CoT)
Think step by step. Simple, effective, but linear — can't backtrack.

A → B → C → D

Plan-then-Execute
Generate a full plan upfront, then execute. Good for predictable tasks.

[Plan] → A → B → C

ReAct (Interleaved)
Plan one step, observe, re-plan. Best for uncertain tasks.

Plan → A → Obs → Replan

Tree-of-Thought
Explore multiple paths, evaluate each, pick best. Expensive but thorough.

A → { B1, B2, B3 } → best
Each strategy trades off between planning quality and computational cost

MCP: Model Context Protocol
The emerging standard for how agents connect to tools and data sources
Think of MCP as USB-C for AI agents — a universal standard so any agent can connect to any tool.
WITHOUT MCP
Custom integration per tool per agent
N agents × M tools = N×M integrations
Fragile, expensive to maintain
WITH MCP
Standard protocol for tool discovery
N agents + M tools = N+M integrations
Plug-and-play tool ecosystem
Adopted by: Anthropic, OpenAI, Google, Cursor, Replit, and growing

Memory & State Management
Agents without memory are stateless — and severely limited

Scratch Pad  (Within a task)
Working memory for the current reasoning chain.
Implementation: System prompt + conversation history

Short-Term  (Within a session)
Conversation context across multiple turns.
Implementation: Sliding window of messages

Long-Term  (Across sessions)
Persistent knowledge about users, past interactions.
Implementation: Vector DB + retrieval (RAG)

Human-in-the-Loop Patterns

Approval Gates
Agent proposes action, human approves before execution
High-stakes: refunds, account changes, escalations

Confidence Thresholds
Agent acts autonomously when confident, asks human when uncertain
Mixed workloads — easy auto, hard escalate

Audit Trails
Agent acts freely but every action logged for review
High-volume, low-risk needing compliance

Escalation Triggers
Predefined rules that force human handoff
Angry customer, legal language, high-value
Agent proposes, human disposes — until trust is earned through data

Step 2: System Prompt Design

You are a customer support agent for TechStore.

RULES:
- Always look up the order before answering questions about it
- Never make up order information — use the lookup_order tool
- Escalate to human if: customer is angry, legal issue,
  or you cannot resolve in 3 turns
- Be empathetic but concise
- Maximum 5 tool calls per conversation
Notice: The system prompt includes guardrails (max tool calls, escalation rules, no hallucination). First line of defense.

Step 3: The ReAct Loop

def agent_loop(user_query, max_iters=5):
    messages = [system_prompt, user_query]

    for i in range(max_iters):
        response = gemini.generate(
            messages, tools=TOOLS
        )

        if response.has_function_call:
            result = execute(response.fn)
            messages.append(result)
        else:
            return response.text  # Done!

    return 'Max iterations reached'

Max iterations
Prevents infinite loops

Message history
Scratch pad memory

Tool detection
Check for function calls

Execution bridge
Your code runs the function

Graceful exit
Return text or error
