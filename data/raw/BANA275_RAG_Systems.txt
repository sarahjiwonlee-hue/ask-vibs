Why Chunking Matters
Documents are too large to fit in LLM context — we must break them down

Too Small
• Lost context
• Fragmented meaning
• More chunks to search

Just Right
• Complete thoughts
• Semantic coherence
• Efficient retrieval

Too Large
• Noise in retrieval
• Diluted relevance
• Higher token costs
Chunking strategy directly impacts retrieval quality and answer accuracy
10
Chunking Strategies

Fixed-Size
Split by character/token count
✓ Simple
✗ Breaks sentences

Recursive
Split hierarchically (¶→sent→word)
✓ Respects structure
✗ More complex

Semantic
Split by meaning boundaries
✓ Best coherence
✗ Needs NLP

Document-Aware
Split by headers/sections
✓ Preserves structure
✗ Format-dependent

Agentic
LLM decides boundaries
✓ Most intelligent
✗ Slow, expensive
Start with recursive chunking — it works well for most use cases
11
Fixed-Size Chunking
Split text into equal-sized pieces by character or token count

The quick brown fox jumps over the lazy dog. The dog was sleeping peacefully.
↓ Split every 30 characters

"The quick brown fox jump"

"s over the lazy dog. The"

" dog was sleeping peace"

def fixed_chunk(text, size=500, overlap=50):
    chunks = []
    for i in range(0, len(text), size - overlap):
        chunks.append(text[i:i + size])
    return chunks
Problem: Breaks words and sentences mid-way → poor semantic coherence
12
Recursive Chunking
Split hierarchically: paragraphs → sentences → words

Step 1: Try splitting by "\n\n" (paragraphs)
↓ If chunks still too large...
Step 2: Try splitting by ". " (sentences)
↓ If chunks still too large...
Step 3: Split by words

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=50,
    separators=["\n\n", "\n", ". ", " ", ""]
)
chunks = splitter.split_text(document)
Best default choice — respects natural text boundaries

Semantic Chunking
Split where meaning changes — using embeddings to detect boundaries

Sent 1    Sent 2    Sent 3  |  Sent 4    Sent 5
 [0.92]    [0.89]    [0.91] |   [0.45]    [0.88]
                            ↑
               Semantic boundary detected!

def semantic_chunk(sentences, embeddings, threshold=0.7):
    chunks, current = [], [sentences[0]]
    for i in range(1, len(sentences)):
        sim = cosine_sim(embeddings[i-1], embeddings[i])
        if sim < threshold:  # Topic changed
            chunks.append(' '.join(current))
            current = []
        current.append(sentences[i])
    return chunks
Best coherence but requires embedding computation during chunking

What are Embeddings?
Converting text into numerical vectors that capture meaning

"How do I reset my password?"
↓ Embedding Model
[0.023, -0.156, 0.892, ...]
→

Similar meaning = Similar vectors
"I forgot my login credentials"
has a vector close to the query
→ High similarity score!
Key Properties:
• Semantic similarity = vector distance
• Typical dimensions: 768 - 3072 (more = more nuance)
• Once embedded, text can be searched mathematically

Similarity Search

Cosine Similarity
Angle between vectors
Range: -1 to 1
Most common
cos(θ) = A·B / ||A||||B||

Dot Product
Projection
Faster compute
Affected by magnitude
A · B = Σ(ai × bi)

Euclidean
Straight-line distance
Smaller = similar
Dense embeddings
d = √Σ(ai - bi)²
Cosine similarity is the default for most RAG applications

Retrieval Strategies

Dense Retrieval
Uses embeddings

✓ Semantic matching
✓ Handles synonyms
✗ Misses exact terms

Sparse Retrieval
Uses BM25/TF-IDF

✓ Exact keyword match
✓ No ML required
✗ Misses semantics

Hybrid Search
Combines both!

✓ Best of both worlds
✓ Higher accuracy
✗ More complex

Hybrid: score = α × dense + (1-α) × sparse  |  Typical α = 0.5-0.7
Recommendation: Use hybrid search for production

Why Re-rank?
• Bi-encoders (embeddings) are fast but less accurate
• Cross-encoders see query + doc together → much more accurate
• Cross-encoders too slow for entire corpus
• Solution: bi-encoder for recall, cross-encoder for precision
Popular: Cohere Rerank, ColBERT, BGE-Reranker

How Many Chunks to Retrieve?
The top-k parameter: balancing context vs noise

Too Few (k=1-2)
• May miss info
• Fragile
• Good for facts
• Lower cost

Just Right (k=3-5)
• Good coverage
• Manageable noise
• Works for most
• Balanced

Too Many (k=10+)
• Introduces noise
• Dilutes relevance
• Higher cost
• Confuses LLM

Dynamic k strategies:
• Similarity threshold: Only include chunks > 0.7 similarity
• Adaptive: Start k=3, add more if confidence low

Ensuring Faithfulness

The Problem:

Explicit Instructions:
"Use ONLY the provided context"

Quote Requirement:
"Include direct quotes for claims"

Confidence Scoring:
"Rate confidence: High/Med/Low"

Self-Check:
"Verify each claim against sources"

Output Validation:
Post-process to check claims

Memory in RAG Systems
Maintaining context across conversations

Short-Term Memory
• Current conversation context
• Recent Q&A pairs
• Session preferences
• Sliding window of messages
• Cleared after session

Long-Term Memory
• User preferences/history
• Past interactions (summarized)
• Learned user facts
• Stored in vector DB
• Persists across sessions

When to forget: Privacy requests, outdated info, conversation reset
Principle: Remember what helps, forget what doesn't

Security Risks in RAG
New attack surfaces introduced by retrieval

Prompt Injection via Docs
Malicious instructions hidden in knowledge base

Data Poisoning
Attackers upload harmful content

Information Leakage
Sensitive data exposed through retrieval

Access Control Bypass
Users retrieving unauthorized docs

Citation Manipulation
Fake sources add false credibility
Your knowledge base is now part of the attack surface
35
Defense Strategies

Input Layer
• Validate queries
• Rate limiting
• Authentication

Retrieval Layer
• Access control
• Content filtering
• Source trust scoring

Generation Layer
• Clear boundaries
• Output filtering
• PII detection

Monitoring
• Log all queries
• Anomaly detection
• Regular audits
Defense in depth: No single layer is sufficient
36
RAG Security Best Practices

1. Treat chunks as untrusted
Never let retrieved content override system instructions

2. Use delimiters
Clearly separate system prompt from retrieved context

3. Implement access control
Filter retrieval results by user permissions

4. Validate sources
Only ingest documents from trusted origins

5. Monitor continuously
Track unusual query patterns and responses

6. Test adversarially
Red-team your system before deployment
Security is a requirement, not a feature

Why RAG Evaluation is Hard
Multiple components, each can fail differently

Retrieval Failures
Right answer exists but wasn't retrieved

Generation Failures
Right chunks retrieved but LLM got it wrong

Chunking Failures
Answer split across chunks, incomplete

Ranking Failures
Right chunk retrieved but ranked too low

Faithfulness Failures
LLM hallucinated beyond retrieved content
Evaluate both retrieval AND generation separately
