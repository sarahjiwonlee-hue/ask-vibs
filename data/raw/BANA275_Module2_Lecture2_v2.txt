Preprocessing Basics
PREPROCESSING
Tokenization: turning text into units
Goal: convert raw text → tokens (words, subwords, characters)
For BoW/TF‑IDF today: word-level tokens are enough
For LLMs: subword tokenization (BPE / WordPiece) is standard
Tokenization choices change your feature space
PREPROCESSING
Normalization: make text comparable
Lowercasing (usually helpful for BoW/TF‑IDF)
Whitespace normalization, Unicode normalization
Handling punctuation: sometimes signal (e.g., “!!!”), sometimes noise
Numbers: keep when meaningful (prices, dates, IDs)
PREPROCESSING
Stopwords: remove with care
Classic approach: remove high-frequency function words
But negations matter: “not”, “never”, “without” can flip meaning
Rule: start minimal, measure impact, then decide
PREPROCESSING
Stemming vs lemmatization
Stemming: crude chopping (faster, can be messy)
Lemmatization: dictionary-based (cleaner, slower)
In many modern pipelines: optional — TF‑IDF + bigrams often beats heavy preprocessing
Rule: if you do it, justify with evaluation
PREPROCESSING
Preprocessing: when it helps vs hurts
Helps: noisy short text, lots of typos, boilerplate, templated signatures
Hurts: removing negation, stripping domain jargon, losing numbers
The only safe principle: change one thing, measure, keep a log

Bag of Words
BOW
Bag of Words (BoW): the core idea
Document = vector of token counts
Simple, sparse, surprisingly strong
Works well for intent/sentiment/topic in many business datasets
Word ordering does not matter (Bag of Words)
BOW
Toy example: 3 docs → counts
Doc1: “refund request not processed”
Doc2: “cancel subscription immediately”
Doc3: “love this product great service”
Counts become features; the model learns weights
BOW
Sparse vectors: why most entries are zero
Vocabulary can be 10k–100k+ tokens
Each document uses only a tiny fraction of that vocabulary
Sparse representations are efficient with linear models
BOW
What BoW captures well
High-signal keywords: “refund”, “broken”, “fraud”, “cancel”
Intent classification and routing
Fast baselines for product decisions
BOW
What BoW misses
No semantics: synonyms are unrelated features
Weak word order: “not good” problem
Brittle under language drift and paraphrase
BOW
Minimal cleaning rules (today)
Start with lowercasing and whitespace normalization
Avoid aggressive stopword removal (negation matters)
Don’t over-engineer — measure, then adjust

TF‑IDF
TF‑IDF
Counts aren’t enough: why weighting helps
Common words dominate counts but add little signal
We want terms that are informative, not just frequent
TF‑IDF
TF: term frequency (in one line)
How often a term appears in a document
Works, but favors long docs and common words
TF‑IDF
IDF: inverse document frequency (in one line)
Down-weights terms that appear in many documents
Up-weights rarer terms that discriminate classes
TF‑IDF
TF‑IDF intuition (no math)
High weight if: important in this document AND rare overall
Often improves baselines dramatically with little effort
IDF weight
dft is the document frequency of t: number of documents that contain t
dft is an inverse measure of the informativeness of word t

n‑grams
N‑GRAMS
n‑grams: phrase-level signal
Unigram: single token; bigram: token pairs; trigram: triplets
Adds local order without full parsing
Often a big jump for sentiment and intent
Bag of words model
Problem 1: Don’t consider the ordering of words in a document

Order Information: Ngrams!
Unigram: Each component of the vector is a word
Same as Bag of Words model



Bi-grams: Each component of the vector is a “pair of words”




N-grams: Each component of the vector is a “n-tuple of words”
	3= Trigram, 4=Fourgram, etc.

N‑GRAMS
Why bigrams matter (business examples)
“not working” vs “working”
“cancel subscription” vs “subscription”
“late delivery” vs “delivery”
N‑GRAMS
Tradeoff: feature explosion
More n‑grams → more features → more compute and overfitting risk
Control with max_features, min_df, max_df
N‑GRAMS
Practical defaults
Start: ngram_range=(1,2)
max_features: 20k–50k for class
min_df: 1–2 to drop rare noise
5
Baselines
BASELINES
Baseline principle
Ship the simplest thing that works
A strong baseline is a benchmark and a business artifact (ROI, feasibility)
BASELINES
Why Logistic Regression is a default winner
Excellent with high-dimensional sparse features
Fast training, stable, easy to deploy
Coefficients give quick interpretability
BASELINES
Regularization (intuition)
Prevents overfitting on rare tokens and noise
Keeps weights stable and improves generalization
You’ll tune ‘C’ (strength) in the micro-challenge
BASELINES
Alternatives (mention only)
Naive Bayes: very fast, sometimes strong on small data
Linear SVM: strong but less probabilistic
Today: we ship LogReg for consistency
6
Evaluation Discipline
EVALUATION
Demo vs deployable
A demo is one example; deployable means consistent performance
Evaluation makes progress measurable and repeatable
EVALUATION
Train / Validation / Test
Train: fit the model
Validation: tune parameters
Test: final estimate (touch once at the end)
EVALUATION
Split strategies (rules of thumb)
Random split: default for i.i.d. data
Time split: if language drifts over time
Group split: if multiple rows belong to same user/account
EVALUATION
Metrics: accuracy vs F1
Accuracy can mislead with imbalance
Macro F1 treats classes equally
Confusion matrix shows error types
EVALUATION
Confusion matrix: read it fast
TP/FP/FN/TN — connect to business costs
False negatives often hurt more in risk and support scenarios
EVALUATION
Leakage: the silent killer (1)
Exact duplicates or near-duplicates across splits
Symptom: unrealistically high scores
EVALUATION
Leakage: the silent killer (2)
Template leakage: the label is implied by boilerplate text
Metadata leakage: fields that wouldn’t exist at prediction time
EVALUATION
Failure Gallery: your improvement backlog
Collect 10 wrong predictions
Categorize (negation, sarcasm, ambiguity, jargon, tail cases, missing context)
Write a hypothesis: ‘what might fix this?’
