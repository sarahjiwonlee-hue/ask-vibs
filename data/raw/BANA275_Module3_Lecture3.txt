Dividing a vector by its L2 norm makes it a unit (length) vector

What is an embedding?
A learned representation that compresses meaning into numbers
Same model maps many texts into the same coordinate system
The goal: preserve semantic relationships
Visual: “embedding pipeline diagram” (search: embedding model diagram)
INTUITION
Why vectors can encode meaning
Meaning shows up through usage patterns (context)
Similar contexts → similar vectors
Works well for paraphrases + synonyms
Breaks on rare entities unless the model has seen them
BASELINES
TF-IDF vs embeddings (what changes?)
TF-IDF: exact tokens + weights
Embeddings: semantic similarity (paraphrase-friendly)
TF-IDF can win on exact matches and identifiers
Best practice: keep both in your toolkit
EMBEDDINGS
Word vs sentence vs document embeddings
Word: “bank” ≠ “bank” (river vs finance) problem
Sentence/document: captures meaning of the full unit
Today’s choice: short docs → document-level vectors
Later: long docs → chunking + retrieval + reranking

GEOMETRY
Neighborhoods in embedding space
Similar concepts form clusters (“fees”, “expense ratio”, “costs”)
Different topics form separated regions (“tax”, “dividends”, “retirement”)
Visual: “embedding clusters scatterplot illustration” (search: embedding space clusters)
WHEN IT WORKS
Why embeddings beat keywords
Synonyms: “fees” ≈ “charges” ≈ “expense ratio”
Paraphrase: “avoid penalties” vs “no early withdrawal penalty”
Typos and variants
Intent similarity (“compare”, “choose”, “difference between”)
WHEN IT FAILS
When embeddings struggle
Entities/tickers: “VOO”, “QQQ”, “SPY”
Numeric constraints: “$200/month”, “10-year horizon”
Very short queries with little context
Highly domain-specific jargon not seen in training
PRACTICAL
Choosing an embedding model (today)
Lightweight model for speed (class setting)
Normalize embeddings for cosine retrieval
Better models exist, but iteration speed matters
Principle: baseline first, then upgrade

RETRIEVAL
Nearest neighbors: the retrieval primitive
Given a query vector, find the closest document vectors
Return top-k results
This powers semantic search, matching, recs, and clustering
Visual: “kNN diagram” (search: k nearest neighbors)
TOP-K
Why we return top-k (not just top-1)
Top-1 may be wrong; top-k gives options
Recall@k captures “did we include something relevant?”
UI can show top results + allow refinement
Retrieval is a ranking problem
INDEXING
Why we need a vector index
Brute force search is slow at scale
Indexes accelerate nearest neighbor search
Today: simple indexes that run in Colab
Production: approximate search + filters + rerankers
PIPELINE
Retrieval pipeline (end-to-end)
1) Choose corpus (finance posts)
2) Embed corpus (once)
3) Build index
4) Embed query
5) Retrieve top-k and inspect results
PRODUCT THINKING
The “retrieval contract”
Input: user intent in natural language
Output: relevant evidence candidates (not final truth)
Must be measurable (Recall/MRR)
Must be debuggable (failure categories)
EVAL
Recall@k (did we include something relevant?)
Recall@5 = 1 if any relevant doc appears in top 5
High Recall@k is necessary for good UX
But Recall alone ignores ranking quality
You’ll compute Recall@5 today
EVAL
MRR@k (how high did the first relevant appear?)
MRR rewards relevant results appearing near the top
Users rarely read beyond top 1–3
Improving MRR often improves perceived quality
You’ll compute MRR@5 today

Pattern 2 — Routing and triage
Classify by similarity to known categories (“tax”, “retirement”, “fees”)
Route to the right workflow/agent/human
Use metadata filters (account type, jurisdiction)

PATTERNS
Pattern 3 — Duplicate detection
Find near-duplicate questions/posts
Reduce repeated work; improve knowledge bases
Combine with human review for safety
Works well even when wording differs
PATTERNS

Cluster questions by theme (“dividends”, “taxes”, “retirement”)
Identify top drivers of confusion
Create better content/UX flows
Visual: “cluster bubbles” (search: text clustering visualization)

Recommend “related questions” or “next best articles”
Great for cold-start (no behavior data needed)

Evaluate with offline ranking metrics
